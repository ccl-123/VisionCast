## 一、 核心交付目标与代码规范

### 1. 交付目标
* **音视频同步管线**：打通视频 V4L2 采集（13855 MIPI 摄像头优先，USB C270 备用）、RGA 格式转换与缩放、MPP 硬件编码、ALSA（NAU8822）音频采集与 G.711/AAC/PCM 编码的完整管线。
* **画面本地显示（非阻塞）**：在不阻塞和影响推流管线的前提下，将采集到的视频帧实时显示在开发板屏幕上。
* **支持三重传输协议且支持配置切换**：
  * **协议一：RTMP 主动推流客户端**。板端作为 Pusher，主动向 PC 主机服务端（IP: `192.168.137.1`）推送 RTMP 数据流。开发时必须参考 SDK 例程 `rk_media_sdk/rkadk/examples/rkadk_rtmp_test.c`。
  * **协议二：RTP + RTSP + UDP 结合协议**。支持以下两种模式：
    * **RTP over UDP 主动推送**：直接向 PC 主机端口发送裸 RTP 包，并生成本地播放所需的 `.sdp` 文件。
    * **RTSP 协议支持**：板端既可以通过 RTSP 服务端（拉流模式，由 PC 主机连接 `rtsp://<板端IP>:8554/live` 拉流，参考 SDK 例程 `rkadk_rtsp_test.c`）提供数据，也可以支持 RTSP 客户端推流（Push 模式，发送 ANNOUNCE 建立通道）。
  * **协议三：WebRTC 主动推流**。支持以 WHIP (WebRTC HTTP Ingestion Protocol) 协议将音视频流推送至 WebRTC 流媒体服务器（如 MediaMTX / SRS），实现网页端极低延迟的 WebRTC 直接播放。
* **极低延迟**：局域网内端到端延迟控制在 500ms（首版）乃至 200~300ms 以内，支持推流至 PC 端后用播放器或服务器拉流验证。
* **零崩溃与极佳稳定性**：代码能够连续稳定运行 30 分钟以上，无内存泄漏，无死锁，能够优雅地响应并处理信号（如 SIGINT）进行安全退出。

### 2. 代码规范与注释要求
* **C++17 绝对规范**：所有源文件必须遵循 C++17 标准，采用清晰的变量、类命名，遵循面向对象/模块化设计。
* **底层返回值校验**：在涉及 ioctl 调用、V4L2 配置、ALSA 开启、MPP 配置及内存映射 (mmap) 时，必须添加严密的返回值校验，一旦失败需输出明确的诊断日志。
* **详尽的核心注释**：
  * 对所有关键逻辑（V4L2 缓冲区出队/入队、RGA 转换参数、MPP 编码配置参数、RTMP 推流、RTSP 服务/客户端、RTP打包、线程同步与丢帧逻辑、本地渲染逻辑）进行**逐行或关键行注释**。
  * 头文件暴露的类成员函数必须拥有清晰的参数与功能说明。

## 二、 Git 提交规范与里程碑工作流 (Git Commit Workflow)

为了使代码演进历史清晰可追溯，**必须遵循“每完成一个大功能进行一次 Git Commit”的原则**。禁止将所有改动一次性提交。提交信息必须使用**详细、规范的中文**进行编写。

### 1. 提交信息格式 (Commit Message Format)
采用约定式提交 (Conventional Commits) 格式，使用中文书写：
```text
<类型>(<影响范围>): <简短描述>

<详细变更原因与实现细节，说明做了什么、为什么做以及如何实现的>
```
* **类型 (Type)**：
  * `feat`：新增功能
  * `fix`：修复 Bug
  * `build`：构建系统或外部依赖变更 (如 CMakeLists)
  * `refactor`：重构代码 (既不修复 Bug 也不新增功能)
* **影响范围 (Scope)**：说明修改的模块，如 `v4l2`、`mpp`、`rga`、`alsa`、`rtmp`、`rtsp`、`pipeline`、`cmake` 等。

### 2. 建议的里程碑提交节点划分
开发过程中，请按以下大功能块划分节点，编译测试通过后分别提交 Commit：

* **Commit 1 (视频采集与 RGA 转换)**
  * **范围**：`feat(video-capture)`
  * **内容**：实现 V4L2 采集与双摄像头自动回退，集成 RGA 转换 NV12。
* **Commit 2 (本地渲染与非阻塞显示)**
  * **范围**：`feat(display)`
  * **内容**：创建 `DisplayRenderer` 异步线程，实现本地屏幕无阻塞画面渲染渲染。
* **Commit 3 (MPP 硬件视频编码)**
  * **范围**：`feat(mpp-encoder)`
  * **内容**：实现 MPP 初始化、Baseline H.264 编码配置及低延迟调优。
* **Commit 4 (ALSA 音频采集与简易编码)**
  * **范围**：`feat(audio)`
  * **内容**：集成 ALSA `hw:1,0` 音频捕获及音频极低消耗编码。
* **Commit 5 (三重传输协议实现与音视频同步)**
  * **范围**：`feat(transport)`
  * **内容**：实现 RTMP 客户端推流模块、WebRTC (WHIP) 推流模块与 RTP/RTSP/UDP 传输链路实现，支持配置切换，基于 `CLOCK_MONOTONIC` 实现 PTS 音视频时间戳同步。
* **Commit 6 (系统管线整合与防积压主动丢帧控制)**
  * **范围**：`feat(pipeline)`
  * **内容**：整合 `PipelineManager` 双流水线，实现 BlockingQueue 溢出主动丢弃老帧策略，完成端到端低延迟闭环。
* **Commit 7 (CMake 链接与库打包发布脚本)**
  * **范围**：`build(cmake)`
  * **内容**：更新 CMakeLists.txt 并修改 `cross_build.sh` 脚本，自动提取 `librga.so`、`librockchip_mpp.so` 以及相关依赖库打包输出至 `install/visioncast/lib/`。

## 五、 详细开发任务与各模块设计要求

### 任务 1: 视频采集模块 (`VideoCapture`)
* 使用标准 V4L2 API，在子线程内非阻塞轮询出帧（MMAP 方式，默认 4 个 Buffer 缓冲区）。
* 优先读取 13855 MIPI 摄像头（主节点输出 `NV12` 或 `YUYV`）；如果主设备无法点亮，平滑降级至备用摄像头（如 USB C270，抓取 `YUYV` 画面）。

### 任务 2: 图像预处理与本地显示 (`RgaProcessor` & `DisplayRenderer`)
* **RGA 转换**：引入 `im2d.h`，利用 RGA 硬件加速将捕获的帧转换为编码器必须的 `NV12` 色彩空间，并根据配置缩放分辨率（例如统一缩放到 `1280x720`）。
* **非阻塞异步本地渲染显示**：
  * 创建独立的 `DisplayRenderer` 模块，引入单独的显示线程读取待显示图像，使用 OpenCV 窗口界面在本地屏幕上实时输出视频画面。
  * **非阻塞要求**：渲染线程绝不能阻塞视频捕获和推流主链路，以确保推流端 30 FPS 频率不受渲染绘制速率的影响。

### 任务 3: 视频 MPP 低延迟编码 (`MppEncoder`)
* 引入 MPP `rk_mpi.h` API，配置 H.264 硬件编码通道：
  * **画质档次**：`Baseline Profile`（绝对禁用 B 帧，消除重排延迟）。
  * **码率控制**：`CBR`（固定码率控制，如 4Mbps，防止数据突变造成网络堆积）。
  * **GOP 与帧率**：设为 30，GOP 设为 30（1秒一个I帧，拉流端可快速显示首帧）。
  * **低延迟模式**：配置 MPP 输出非阻塞，消除内部缓冲，数据即到即出。

### 任务 4: ALSA 音频采集与简易编码 (`AudioCapture` & `AudioEncoder`)
* 引入 `<alsa/asoundlib.h>`，初始化 ALSA 捕获设备 `hw:1,0` (nau8822 Codec)。
* 配置为采样率 `48000Hz`、采样格式 `S16_LE`、采集周期 `20ms`，配合推流层将采样帧进行相应音频封装与编码。

### 任务 5: 严格时间戳同步与时钟 (`MediaClock`)
* 统一使用单调时钟 `clock_gettime(CLOCK_MONOTONIC, &ts)`（单位为微秒）。
* 在视频 V4L2 驱动出队、音频 ALSA 读出时**瞬间打上原始时间戳**，并在整个转换、编码和推流封装传输阶段一路透传，保证音视频的极佳同步（AV Sync）。

### 任务 6: 三重传输协议推流设计 (`RtmpPusher` & `RtspServer`/`UdpSender` & `WebRtcPusher`)
根据配置加载，实现以下三种完全独立的网络传输分支：
* **分支一：RTMP 主动推流客户端**：
  * 引入 `rtmp_pusher.h`，实现主动建立 RTMP 连接的功能。
  * 参考例程 `rkadk_rtmp_test.c`，使用 RTMP 协议将视频 H.264 编码帧与音频帧封装打包，持续主动推送到配置中的 `rtmp_url`（通常位于主机端 `rtmp://192.168.137.1:1935/live/stream`）。
* **分支二：RTP + RTSP + UDP 结合传输方案**：
  * **模式 A（RTP over UDP 推流）**：利用 `udp_sender.h`，直接将视频 RTP 包和音频 RTP 包分别向配置中主机的 UDP 端口（例如：视频 5004，音频 5006）主动发送，并在安装根目录下自动生成一份供主机端播放器直接双击使用的 `test.sdp` 文件。
  * **模式 B（RTSP 服务端拉流）**：利用 `rtsp_server.h`，在板端运行一个轻量级的 RTSP 流媒体服务端（默认监听 `8554` 端口）。参考例程 `rkadk_rtsp_test.c` 开启 RTSP 服务，支持 PC 播放端主动连线拉流。
  * **模式 C（RTSP 客户端主动推流，可选）**：板端作为 RTSP 客户端向主机端 RTSP 服务端主动推送数据包。
* **分支三：WebRTC WHIP 主动推流**：
  * 引入 `webrtc_pusher.h`，支持通过 WebRTC WHIP (WebRTC HTTP Ingestion Protocol) 协议推流。
  * 板端向配置中的 `webrtc_url`（例如 `http://192.168.137.1:8889/live/stream`）发起 HTTP POST 进行 SDP 交换。
  * 握手成功后建立 PeerConnection，通过 ICE、DTLS 及 SRTP 进行加密打包传输音视频数据。

### 任务 7: 流水线调度与防抖丢帧策略
* **主动丢帧机制**：为视频和音频通道建立限定最大大小的环形阻塞队列（`BlockingQueue`），如视频缓冲队列最大容量设为 2。
* **丢帧策略**：如果网络发送过慢导致队列溢出，**必须立刻丢弃老帧**，永远保持发送最新的实时画面。

### 任务 8: 性能监控与统计 (`PerfMonitor`)
* 根据 `docs/performance_test.md` 定义的格式输出 FPS、RGA 耗时、MPP 编码耗时、实际推流码率等信息。
