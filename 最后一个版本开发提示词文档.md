# VisionCast 最后一个版本开发与集成说明文档 (RK3588 硬编、低时延与多协议传输一体化已完工版)

本文档详细总结了 **VisionCast 智能眼镜音视频实时低延迟推流系统** 最后一个版本的全部核心设计指标、集成交付状况与各模块在 RK3588 板端的最终实现方案。本版本已实现全部音视频采集、预处理、硬件编码、本地画面非阻塞显示以及四重传输协议推流。

---

## 一、 核心交付目标完成情况

系统在 ELF-RK3588 板端已完成完整闭环联调，核心交付指标均已达到：

1. **音视频同步管线打通**：
   - 打通了 V4L2 视频采集（主节点 13855 MIPI 摄像头，备用节点 USB C270）、RGA 格式色彩空间转换、MPP 硬件 H.264 编码。
   - 整合了 ALSA（nau8822 Codec，`hw:1,0`）音频采集与 G.711A (PCMA) / AAC 编码的并行流水线。
2. **画面本地显示（完全非阻塞）**：
   - 成功将本地预览渲染剥离至独立的异步线程 `DisplayRenderer`。基于 OpenCV 实现屏幕画面的实时显示（运行在 X11/XWayland 环境下）。
   - 实测证明，渲染线程不会因窗口事件或绘制时延对视频捕获、编码及推流的 30 FPS 主链路引入任何阻塞，极大地提高了推流稳定性。
3. **支持四重传输协议且支持配置切换**：
   - **协议一：RTMP 主动推流客户端**。板端作为 Pusher，通过动态链接 FFmpeg 动态加载库方式，将 H.264/AAC 打包推送至目的服务器。
   - **协议二：RTSP 服务端监听**。板端利用 `librtsp.a` 在 `8555` 端口监听并推流，免去配置外部 RTSP 转发服务器。
   - **协议三：RTP over UDP 主动推流**。板端作为 Pusher 向主机端口发送音视频 RTP 包，自动在本地生成播放所需的 `test.sdp` 描述文件，并具备空挂断容错。
   - **协议四：WebRTC WHIP 主动推流**。以 WHIP 协议推送至 WebRTC 流媒体服务器（如 MediaMTX / SRS），通过本地回环地址绑定及 SDP Offer 角色替换，彻底攻克了 DTLS 握手超时问题。
4. **低时延与稳定性**：
   - 局域网/环回端到端延迟控制在 **200~300ms**。
   - 经过长期运行压力测试，系统无死锁、无内存泄漏，并能优雅地捕获 `SIGINT` 信号安全释放 MPP/RGA/ALSA/V4L2 等全部硬件描述符与上下文。

## 二、 已完工的 C++17 关键模块与任务实现

### 任务 1: 视频采集模块 (`VideoCapture`)
- 封装了标准 V4L2 API。在子线程内非阻塞式通过 `epoll` / `select` 轮询出帧（MMAP 方式，默认申请 4 个 Buffer 帧缓冲区）。
- 支持双摄像头平滑退避。默认拉取 13855 MIPI 摄像头（输出 NV12），若出错则平滑退降至备用摄像头（USB C270，抓取 YUYV 画面）。

### 任务 2: 图像预处理与本地显示 (`RgaProcessor` & `DisplayRenderer`)
- **RGA 转换**：封装 `im2d_api` 库，调用 RGA 硬件加速单元将 V4L2 抛出的 YUYV / NV12 帧迅速缩放到 `1280x720`，并统一执行色彩空间转换输出编码器所需的 NV12 数据。
- **异步预览**：`DisplayRenderer` 内部开辟独立渲染线程，利用 OpenCV HighGUI 处理显示。

### 任务 3: 视频 MPP 低延迟编码 (`MppEncoder`)
- 封装 MPP MPI API，构建 H.264 硬编码管道：
  - **画质档次**：配置为 `H264_PROFILE_BASELINE`（绝对禁用 B 帧，从底层消除重排带来的多帧缓冲时延）。
  - **码率控制**：采用 `CBR` 固定码率分配（如 4Mbps），避免网络发包突变。
  - **低延迟参数**：将 GOP 设为 30，开启 MPP 内部低延迟模式，视频帧随到随编，即刻输出。

### 任务 4: ALSA 音频采集与编码 (`AudioCapture` & `AudioEncoder`)
- 封装 `asoundlib`，开启 nau8822 音频捕获（`hw:1,0`）。
- 采样率 `48000Hz`，采样格式 `S16_LE`，采集周期 `20ms`。在 ALSA 硬件层拒绝单声道时以双声道采集并进行均值混音输出单声道，以适配推流分发。

### 任务 5: 严格时间戳同步与时钟 (`MediaClock`)
- 音视频采集和传输管线统一基于 `CLOCK_MONOTONIC` 单调时钟进行微秒级计时。
- 视频在 V4L2 驱动出队、音频在 ALSA 读出时瞬间打上原始时间戳，透传至编码、打包与网络发送模块，确保播放端音画极佳同步。

### 任务 6: 网络传输协议模块 (`AvTransport`)
- **RTMP**：以 C 接口形式 dlopen 加载 `libavformat` 和 `libavcodec`，打包流推送至目标服务器。
- **RTSP**：利用静态链接 `librtsp.a` 服务端分发流。
- **RTP over UDP**：向指定的 IP/端口发送裸 UDP 包，并生成对应的 `test.sdp`。修改了 UDP socket 对 `ECONNREFUSED` 的拦截逻辑，确保播放端不在线时系统不会崩溃。
- **WebRTC WHIP**：链接 `libdatachannel` 建立 PeerConnection。实现本地回环 IP 绑定 (`config.bindAddress`) 与 Offer 中的 `a=setup:active` 强控重写，彻底解决了 Pion/MediaMTX 环境下的 DTLS 握手超时问题。

### 任务 7: 丢帧策略与防堆积
- 音视频通道建立限定最大大小的环形阻塞队列（`BlockingQueue`），如视频缓存队列最大容量设为 2。
- 如果网络发送过慢导致队列溢出，**必须立刻丢弃老帧**，永远保持发送最新的实时画面。

## 三、 CMake 编译与打包规范

### 1. CMake 依赖结构
`CMakeLists.txt` 已整合并配置了 RGA 和 MPP 头文件路径以及其在 ELF-RK3588 环境下的 `.so` 库地址：
- **RGA 头文件**：`rk_media_sdk/linux-rga/im2d_api` 目录。

- **依赖打包**：编译产物和共享库将安装至系统运行目录：`/home/elf/open_project/VisionCast/install/visioncast/`。

### 2. 编译打包命令
在 RK3588 开发板端直接执行本机构建脚本完成编译与打包部署：
```bash
./scripts/build/device_build.sh
```

## 四、 四重推流测试验收说明

在开发板端进行推流验证时，可以通过以下步骤完成验收：

1. **RTMP 模式测试**：
   - 运行 MediaMTX（监听端口 `1936`）。
   - 启动推流：`./install/visioncast/scripts/run/run_usb_camera.sh rtmp`。
   - 拉流验证：`ffplay rtmp://localhost:1936/live/stream`。

2. **WebRTC WHIP 模式测试**：
   - 运行 MediaMTX（监听端口 `8891`）。
   - 启动推流：`./install/visioncast/scripts/run/run_usb_camera.sh webrtc`。
   - 拉流验证：浏览器访问 `http://localhost:8891/live/stream`。

3. **RTSP 模式测试**：
   - 启动推流：`./install/visioncast/scripts/run/run_usb_camera.sh rtsp`。
   - 拉流验证：`ffplay rtsp://localhost:8555/live`。

4. **RTP 模式测试**：
   - 启动推流：`./install/visioncast/scripts/run/run_usb_camera.sh rtp`。
   - 拉流验证：将生成的 `test.sdp` 拖入 VLC，或者在命令行执行 `ffplay -protocol_whitelist file,udp,rtp test.sdp`。
