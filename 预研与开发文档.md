# VisionCast 智能眼镜音视频推流系统 - 预研与开发文档

# ELF-RK3588 平台适配版

> **版本**：v0.3 Pre-development / ELF-RK3588 Adaptation
> **目标平台**：ELF-RK3588 开发板
> **摄像头模块**：13855 MIPI 摄像头，USB C270 作为备用调试输入
> **音频设备**：rockchip-nau8822，ALSA 设备 `hw:1,0`
> **核心目标**：低延迟音视频采集、硬件编码、网络推流、后续支持 DMA-BUF 零拷贝优化
> **最后更新**：2026-06-05

---

## 1. 项目概述

### 1.1 项目背景

VisionCast 是一个基于 ELF-RK3588 开发板的智能眼镜音视频推流系统预研项目。项目目标是模拟智能眼镜第一视角采集场景，实现摄像头视频与麦克风音频的实时采集、编码、封装和网络推流。

ELF-RK3588 平台具备较强的多媒体处理能力，芯片内部集成 MIPI CSI、ISP、RGA、MPP 视频编解码器等硬件单元，适合构建低延迟音视频处理管线。本项目计划复用 Firefly RK3588 SDK 中的 Rockchip 媒体能力，包括 MPP、RGA、RKAIQ、V4L2、ALSA 等组件，并结合 ELF 板端外设进行适配。

与传统 PC 端推流方案不同，本项目关注嵌入式端侧部署，重点验证 RK3588 平台在视频采集、图像预处理、硬件编码和实时推流场景下的性能表现，为后续智能眼镜、远程协作、第一视角直播、边缘 AI 识别等应用提供底层基础。

---

## 1.2 项目目标

本项目的阶段性目标如下：

1. 完成 ELF-RK3588 板端摄像头与音频设备识别；
2. 打通视频采集、音频采集、编码与推流基础链路；
3. 支持 13855 MIPI 摄像头作为正式视频输入；
4. 支持 USB C270 摄像头作为备用调试输入；
5. 支持板载 NAU8822 音频采集；
6. 优先使用 RK3588 硬件编码能力，降低 CPU 占用；
7. 在功能闭环后逐步优化延迟、拷贝次数和线程调度；
8. 后续根据需求支持 DMA-BUF 零拷贝、RGA 硬件预处理、MPP 低延迟编码参数调优；
9. 为后续接入 AI 识别、远程通话、WebRTC 或 RTSP 服务打基础。

---

## 1.3 技术路线调整说明

本项目不强制在首版阶段完全弃用 GStreamer、FFmpeg、Rockit 等封装层。考虑到预研阶段需要快速验证设备链路和推流闭环，开发策略调整为：

```text
先跑通功能
    ↓
再定位瓶颈
    ↓
逐步替换关键路径
    ↓
最终优化为低延迟硬件管线
```

因此，项目采用“分层开发、可替换媒体后端”的策略：

| 阶段  | 技术路线                                    | 目标           |
| --- | --------------------------------------- | ------------ |
| 阶段一 | GStreamer / FFmpeg / v4l2-ctl / arecord | 快速验证采集与推流闭环  |
| 阶段二 | V4L2 + ALSA + MPP                       | 自研基础音视频管线    |
| 阶段三 | V4L2 + RGA + MPP                        | 加入硬件图像预处理    |
| 阶段四 | DMA-BUF + RGA + MPP                     | 减少内存拷贝，降低延迟  |
| 阶段五 | RTSP / RTP / WebRTC                     | 完善网络传输和播放端适配 |

这种路线比“一开始完全去封装化”更稳。封装框架用于验证链路，自研 C/C++ 管线用于性能优化，二者并不冲突。

当前项目存在两类可用视频输入：

| 摄像头            | 类型       | 设备链路                            | 用途   |
| -------------- | -------- | ------------------------------- | ---- |
| 13855 摄像头      | MIPI CSI | RKCIF → RKISP → mainpath        | 正式方案 |
| C270 HD WEBCAM | USB UVC  | `/dev/video21` / `/dev/video22` | 调试备用 |

其中，13855 摄像头作为智能眼镜第一视角采集的正式输入，更符合嵌入式低延迟场景。USB C270 摄像头用于快速测试 V4L2、编码和推流流程。

```text
13855 MIPI Sensor

MIPI CSI-2
    ↓
RKCIF
    ↓
RKISP / RKAIQ
    ↓
RKISP Mainpath
    ↓
RGA / MPP
    ↓
网络推流
```

备用调试链路如下：

```text
USB C270
    ↓
V4L2
    ↓
RGA / 软件转换
    ↓
MPP 编码
    ↓
网络推流
```

## 2.3 音频设备枚举

在 ELF-RK3588 板端执行：

```bash
arecord -l
```

当前输出如下：

```text
**** List of CAPTURE Hardware Devices ****
card 1: rockchipnau8822 [rockchip-nau8822], device 0: dailink-multicodecs nau8822-hifi-0 [dailink-multicodecs nau8822-hifi-0]
  Subdevices: 1/1
  Subdevice #0: subdevice #0
```

因此当前音频采集设备为：

```text
hw:1,0
```

项目首版音频采集建议使用 ALSA PCM 接口，初始参数建议为：

| 参数   | 建议值                 |
| ---- | ------------------- |
| 采样率  | 48000 Hz            |
| 声道数  | 1 或 2               |
| 采样格式 | S16_LE              |
| 采样周期 | 10 ms 或 20 ms       |
| 首版编码 | PCM / G.711 / Opus  |
| 调试命令 | `arecord -D hw:1,0` |

需要注意的是，RK MPP 主要负责视频编解码，不负责通用音频编码。因此文档中不应写成“音频也由 MPP 硬编码”。更准确的表述是：**视频使用 MPP 硬件编码，音频使用 ALSA 采集并根据协议需求选择 PCM、G.711、AAC 或 Opus 编码。**

# 3. SDK 组件选型

## 3.1 主要组件

从 Firefly RK3588 SDK 或板端系统中复用的核心组件如下：

| 组件               | 作用                 | 是否优先使用         |
| ---------------- | ------------------ | -------------- |
| V4L2             | 摄像头采集接口            | 是              |
| Media Controller | MIPI / ISP 拓扑管理    | 是              |
| RKAIQ            | ISP 3A 与图像质量控制     | MIPI 摄像头阶段使用   |
| RGA              | 图像缩放、裁剪、格式转换       | 是              |
| MPP              | H.264/H.265 视频硬件编码 | 是              |
| ALSA             | 音频采集               | 是              |
| GStreamer        | 快速验证采集、编码、推流链路     | 可选             |
| FFmpeg           | 快速封装、推流、播放端调试      | 可选             |
| Rockit / RKADK   | 高层媒体封装             | 预研可参考，核心管线不强依赖 |

---

## 3.2 不再强制去封装化

原方案中写到“弃用 GStreamer/FFmpeg/Rockit 等高层封装，直接调用 MPP/RGA/V4L2 C API”。该表述过于绝对，不适合作为预研阶段约束。

修订后的原则为：

```text
封装框架用于快速验证；
底层 API 用于性能优化；
核心链路保持可替换；
最终根据延迟和稳定性决定保留层级。
```

也就是说：

1. 调试阶段可以使用 GStreamer 快速验证摄像头和推流；
2. 功能阶段可以使用 FFmpeg 或 RTSP Server 快速完成播放端兼容；
3. 性能优化阶段再逐步切换到 V4L2、RGA、MPP 原生接口；
4. 对延迟要求最高的路径才需要做 DMA-BUF 零拷贝优化；
5. 不为了“底层化”而牺牲开发效率和系统稳定性。

# 5. 关键技术设计

## 5.1 视频采集设计

### 5.1.1 13855 MIPI 摄像头链路

正式视频输入优先使用 13855 摄像头，其底层链路为：

```text
13855 Sensor
    → MIPI CSI-2
    → RKCIF
    → RKISP
    → rkisp_mainpath
    → V4L2 Video Node
```

当前已枚举到：

```text
/dev/media0    # rkcif media graph
/dev/media1    # rkisp media graph
/dev/video11 ~ /dev/video17    # rkisp_mainpath 候选节点
```

下一步需要通过以下命令确认具体 Sensor、拓扑和输出节点：

```bash
dmesg | grep -Ei "13855|ov13855|camera|mipi|csi|rkcif|rkisp"

media-ctl -p -d /dev/media0

media-ctl -p -d /dev/media1
```

查询 ISP 输出节点能力：

```bash
for i in {11..17}; do
    echo "===== /dev/video$i ====="
    v4l2-ctl -d /dev/video$i -D
    v4l2-ctl -d /dev/video$i --list-formats-ext
done
```

最终目标是找到支持 NV12、NV16、YUYV 等格式的 mainpath 输出节点，并将其接入 MPP 编码器。

---

### 5.1.2 USB C270 调试链路

USB C270 摄像头用于备用调试，当前节点为：

```text
/dev/video21
/dev/video22
```

建议先查询格式能力：

```bash
v4l2-ctl -d /dev/video21 -D
v4l2-ctl -d /dev/video21 --list-formats-ext

v4l2-ctl -d /dev/video22 -D
v4l2-ctl -d /dev/video22 --list-formats-ext
```

C270 常见输出格式包括 MJPEG 和 YUYV。若选择 YUYV，链路较简单：

```text
YUYV
    ↓
RGA 转 NV12
    ↓
MPP H.264 编码
```

若选择 MJPEG，则需要先进行 MJPEG 解码：

```text
MJPEG
    ↓
JPEG/MJPEG 解码
    ↓
NV12
    ↓
MPP H.264 编码
```

因此，C270 适合调试，不适合作为最终低延迟链路。

---

## 5.2 图像预处理设计

图像预处理主要由 RGA 完成，功能包括：

1. 分辨率缩放；
2. 图像裁剪；
3. 图像旋转；
4. 色彩空间转换；
5. 输入格式转换为 MPP 编码器友好的 NV12。

推荐目标格式：

```text
NV12
```

推荐目标分辨率：

| 场景    | 分辨率                    |
| ----- | ---------------------- |
| 首版调试  | 640×480                |
| 标清低延迟 | 1280×720               |
| 高清推流  | 1920×1080              |
| 后续测试  | 根据 13855 Sensor 输出能力调整 |

对于推流系统，优先考虑稳定帧率和端到端延迟，而不是一开始追求最高分辨率。

---

## 5.3 视频编码设计

视频编码使用 Rockchip MPP。

首选编码格式：

```text
H.264
```

原因：

1. 播放端兼容性最好；
2. RTSP/RTP 支持成熟；
3. 浏览器、播放器、移动端支持广泛；
4. 调试成本低。

后续可增加 H.265：

```text
H.265
```

H.265 压缩率更高，但播放端兼容性和延迟调试复杂度高于 H.264。

### 5.3.1 编码参数建议

首版建议参数：

| 参数      | 建议值             |
| ------- | --------------- |
| 编码格式    | H.264           |
| Profile | Baseline / Main |
| 分辨率     | 1280×720        |
| 帧率      | 30 FPS          |
| GOP     | 30 或 15         |
| 码率控制    | CBR             |
| 码率      | 2 Mbps ~ 6 Mbps |
| B 帧     | 关闭              |
| 低延迟模式   | 开启              |
| 输入格式    | NV12            |

低延迟场景下，应尽量避免 B 帧，并缩短 GOP。GOP 越短，首帧显示和网络恢复越快，但码率会增加。

---

## 5.4 音频采集与编码设计

音频采集使用 ALSA。

设备：

```text
hw:1,0
```

测试命令：

```bash
arecord -D hw:1,0 --dump-hw-params
```

采集测试：

```bash
arecord -D hw:1,0 -f S16_LE -r 48000 -c 2 -d 5 /tmp/audio_test.wav
```

音频编码可按阶段选择：

| 阶段    | 格式    | 说明           |
| ----- | ----- | ------------ |
| 调试阶段  | PCM   | 最简单，延迟低，但带宽高 |
| 语音低延迟 | G.711 | 实现简单，CPU 占用低 |
| 实时通话  | Opus  | 低延迟、抗丢包能力强   |
| 通用播放  | AAC   | 兼容性好，但延迟可能更高 |

首版建议先使用 PCM 或 G.711，后续再增加 Opus 或 AAC。

---

## 5.5 时间戳设计

音视频同步使用统一时钟源。

推荐使用：

```c
clock_gettime(CLOCK_MONOTONIC, &ts);
```

统一转换为微秒：

```c
uint64_t pts_us = ts.tv_sec * 1000000ULL + ts.tv_nsec / 1000;
```

视频帧、音频帧均使用该时间戳系统，避免系统时间变化造成 PTS 跳变。

时间戳原则：

1. 采集时打 PTS；
2. 编码后保留原始采集 PTS；
3. 封包时使用同源时间戳；
4. 不使用系统实时时钟作为媒体时间；
5. 音频和视频都基于 `CLOCK_MONOTONIC`。

# 6. 网络推流方案

## 6.1 可选推流协议

| 协议     | 优点         | 缺点          | 适用阶段 |
| ------ | ---------- | ----------- | ---- |
| UDP 裸流 | 实现简单，延迟低   | 无标准播放端，可靠性差 | 早期调试 |
| RTP    | 实时性好，适合音视频 | 需要封包逻辑      | 推荐   |
| RTSP   | 播放端兼容好     | 服务端实现稍复杂    | 推荐   |
| WebRTC | 超低延迟，浏览器支持 | 实现复杂        | 后续扩展 |
| RTMP   | 生态成熟       | 延迟偏高        | 不优先  |

首版建议采用：

```text
RTP / RTSP
```

原因是 RTSP 便于使用 VLC、ffplay 等工具验证。

---

## 6.2 播放端测试方式

如果使用 RTSP：

```bash
ffplay rtsp://板端IP:8554/live
```

或：

```bash
vlc rtsp://板端IP:8554/live
```

如果使用 UDP/RTP，需要根据实际封包格式指定 SDP 或播放参数。

# 7. 开发路线

## 7.1 阶段一：设备验证

目标：确认摄像头、音频和基础媒体链路可用。

任务：

1. 确认 13855 摄像头驱动加载；
2. 查看 `media-ctl` 拓扑；
3. 确认 rkisp mainpath 输出节点；
4. 测试 `/dev/video11` ~ `/dev/video17` 视频采集能力；
5. 测试 USB C270 备用输入；
6. 测试 `hw:1,0` 音频采集；
7. 记录支持的分辨率、帧率和像素格式。

关键命令：

```bash
dmesg | grep -Ei "13855|ov13855|camera|mipi|csi|rkcif|rkisp"

media-ctl -p -d /dev/media0
media-ctl -p -d /dev/media1

for i in {11..17}; do
    echo "===== /dev/video$i ====="
    v4l2-ctl -d /dev/video$i -D
    v4l2-ctl -d /dev/video$i --list-formats-ext
done

arecord -D hw:1,0 --dump-hw-params
```

---

## 7.2 阶段二：快速推流闭环

目标：使用封装框架快速验证采集、编码和推流。

可选方案：

1. GStreamer 管线；
2. FFmpeg 命令行；
3. Rockchip 示例程序；
4. 简单 RTSP Server 示例。

该阶段不追求最优延迟，只验证：

```text
摄像头能采集
音频能采集
视频能编码
网络能播放
```

---

## 7.3 阶段三：自研基础管线

目标：使用 C/C++ 自研基础媒体管线。

基础链路：

```text
V4L2 Capture
    ↓
RGA Convert
    ↓
MPP Encoder
    ↓
RTP / RTSP Sender
```

音频链路：

```text
ALSA Capture
    ↓
Audio Encoder / PCM
    ↓
RTP / RTSP Sender
```

该阶段重点关注：

1. 模块接口设计；
2. 线程模型；
3. Buffer 生命周期；
4. 时间戳传递；
5. 编码参数调优；
6. 基础异常处理。

---

## 7.4 阶段四：低延迟优化

目标：降低端到端延迟。

优化方向：

1. 使用 MPP 低延迟编码参数；
2. 关闭 B 帧；
3. 缩短 GOP；
4. 控制编码缓存；
5. 控制发送缓存；
6. 减少线程排队；
7. 减少内存拷贝；
8. 使用 DMA-BUF FD 传递图像；
9. RGA 和 MPP 之间尽量避免 CPU 拷贝；
10. 使用实时统计分析每段耗时。

# 8. 线程模型设计

推荐初始线程模型：

```text
VideoCaptureThread
    ↓
VideoProcessThread
    ↓
VideoEncodeThread
    ↓
NetworkSendThread
```

音频线程：

```text
AudioCaptureThread
    ↓
AudioEncodeThread
    ↓
NetworkSendThread
```

统一控制线程：

```text
MainControlThread
    ↓
PipelineManager
    ↓
ConfigManager
    ↓
PerfMonitor
```

后续可根据性能测试结果调整为更少线程或环形队列模型。低延迟场景下，线程并不是越多越好，过多线程会引入调度延迟和缓存积压。

建议每个队列设置最大缓存长度，例如：

```text
Video raw queue: 2 ~ 3 frames
Video encoded queue: 2 ~ 3 packets
Audio queue: 2 ~ 5 packets
```

当队列满时，实时推流系统应优先丢弃旧帧，而不是继续堆积延迟。

# 9. Buffer 与零拷贝策略

## 9.1 初始策略

首版可以使用普通 MMAP Buffer 或用户态 Buffer 完成功能验证。此阶段允许存在少量拷贝，重点是打通链路。

## 9.2 优化策略

在基础链路稳定后，逐步切换到 DMA-BUF：

```text
V4L2 Export DMA-BUF FD
    ↓
RGA Import DMA-BUF FD
    ↓
MPP Import DMA-BUF FD
```

目标是减少 CPU 参与图像拷贝，提高帧率并降低延迟。

## 9.3 工程原则

零拷贝不是首版必须项，而是性能优化项。开发过程中应先保证：

1. 画面能采；
2. 编码正确；
3. 播放稳定；
4. 时间戳正确；
5. 延迟可测。

之后再逐步优化到 DMA-BUF 零拷贝。

# 15. 风险点与注意事项

## 15.1 13855 摄像头适配风险

可能存在以下问题：

1. Sensor 驱动未加载；
2. 设备树未正确配置；
3. I2C 地址不匹配；
4. MIPI lane 数配置错误；
5. reset / pwdn GPIO 配置错误；
6. 外部时钟配置错误；
7. RKAIQ XML 配置缺失；
8. media graph 未正确链接；
9. ISP mainpath 节点不明确。

因此，正式开发前必须先确认 `dmesg` 和 `media-ctl` 输出。

---

## 15.2 音视频同步风险

音视频采集来自不同设备，必须使用统一 PTS。否则推流时间变长后可能出现音画不同步。

解决方案：

1. 统一使用 `CLOCK_MONOTONIC`；
2. 采集时打时间戳；
3. 不使用系统实时时钟；
4. 控制音频缓冲区长度；
5. 控制视频队列长度。

---

## 15.3 低延迟优化风险

低延迟优化不是单点问题，而是完整链路问题。可能产生延迟的位置包括：

1. 摄像头曝光；
2. ISP 处理；
3. V4L2 Buffer 队列；
4. RGA 处理；
5. MPP 编码缓存；
6. RTP/RTSP 发送缓存；
7. 网络传输；
8. 播放端缓冲。

因此，必须分段统计耗时，不能只看编码器耗时。

# 17. 当前结论

VisionCast 项目在 ELF-RK3588 平台上具备可行性。当前系统已经枚举出 Rockchip MIPI 摄像头链路、RKISP 输出节点、USB 摄像头节点以及 NAU8822 音频采集设备，具备开展音视频推流系统预研的基础条件。

项目不应在首版阶段强制完全去封装化。更合理的路线是：

```text
使用封装框架快速验证功能
    ↓
使用 V4L2 / ALSA / MPP 构建自研管线
    ↓
使用 RGA 和 DMA-BUF 优化性能
    ↓
根据实际延迟决定最终架构
```

最终目标是形成一套适配 ELF-RK3588 的低延迟音视频采集、编码和推流系统，为智能眼镜第一视角实时音视频应用提供底层基础。
