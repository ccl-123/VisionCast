> **版本**：v1.0 (Final Version)
> **目标板端**：ELF-RK3588 开发板
> **核心特性**：硬编解码、图像加速、异步显示、多协议低延迟推流
> **更新时间**：2026-06-07

## 1. 项目概述与实现目标

VisionCast 是一个专为嵌入式 RK3588 平台定制的高性能、低延迟音视频采集与推流系统。系统模拟智能眼镜第一视角采集场景，通过硬加速单元实现极速数据处理，提供 WebRTC、RTSP、RTMP、RTP 多协议的平滑切换与推流分发。

### 1.1 已实现的核心功能
1. **非阻塞多源视频采集**：实现 V4L2 视频采集模块。默认开启 13855 MIPI 摄像头（主路径 `/dev/video11`），并支持在异常情况下自动退避至 USB C270 备用摄像头（`/dev/video21`），支持多格式检测与自动适配。
2. **ALSA 硬件音频采集**：利用 ALSA API 从板载 nau8822 音频捕获设备（`hw:1,0`）采集 48kHz S16_LE 原始 PCM，支持通过 JSON 分别配置采集声道数与推流声道数，并在需要时执行双声道到单声道的均值混音。
3. **RGA 硬件加速预处理**：集成 `im2d` 接口，使用 RK3588 RGA 硬件加速单元将原始帧（NV12 或 YUYV）缩放并转换为编码器所期望的 NV12 格式，避免 CPU 参与格式搬运。
4. **MPP 硬件视频编码**：配置并封装 Rockchip MPP 库。对 NV12 图像进行低延迟、高效率的 H.264/H.265 CBR 硬编码，完全禁用 B 帧以消除重排延迟，GOP 设为 30。
5. **异步预览显示**：基于 `DisplayRenderer` 模块开辟了独立于推流主管线的渲染线程，使用 OpenCV 高级 UI 对视频流进行非阻塞式的实时预览显示，确保本地画面渲染不拖慢推流主管线。
6. **多协议网络推流分发**：
   - **WebRTC (WHIP)**：链接 FFmpeg 8.1.1 原生 `whip` 封装器。配置建连阶段 `dtls_active` 状态以强控主动握手角色，并通过编译期补丁实现 candidate 地址与 WHIP URL 宿主机 IP 的智能比对与筛选，彻底消除了多网卡代理及本地回环环境下的建连挂起问题。
   - **RTSP**：链接 FFmpeg 8.1.1 原生 `rtsp` 封装器，向 MediaMTX 等标准服务端发布 H.264/H.265 视频与 Opus 音频，支持 UDP 或 TCP 承载。
   - **RTMP**：直接链接项目内置 FFmpeg 8.1.1，在 H.264/AAC 或 H.265/AAC 规格下向配置的目的 RTMP 服务器进行主动推送。
   - **RTP over UDP**：将音视频 RTP 封包后直接向目标 UDP 端口投递，并在运行时自动在工作目录生成对应的播放描述文件 `test.sdp`。
7. **极低延迟与防抖丢帧**：基于 `CLOCK_MONOTONIC` 实现全管线时间戳同步（A/V Sync）。原始和编码队列使用最大长度为 2 的阻塞队列，当网络拥塞时主动执行丢旧帧策略以维持推流的低时延状态。

## 2. 系统总体设计与数据链路 (Data Pipelines)

VisionCast 系统采用分层模块化设计，将数据流划分为两条独立的管线，并通过 `PipelineManager` 进行综合调度。

### 2.1 视频数据链路
```text
[13855 MIPI CSI / USB C270] (采集源)
            │  (V4L2 MMAP 队列非阻塞出帧)
            ▼
     [VideoCapture] 
            │  (NV12 / YUYV 帧数据)
            ├─────────────────────────────────────────┐
            ▼ (进入推流主管线)                          ▼ (进入本地显示管线)
     [RgaProcessor] (im2d 色彩转换/尺寸缩放)      [DisplayRenderer] (异步线程)
            │  (RGA 加速生成的 NV12)                   │
            ▼                                         ▼
       [MppEncoder] (H.264/H.265 硬编码)         [OpenCV HighGUI] (X11 窗口渲染)
            │  (H.264/H.265 Annex-B 裸码流)
            ▼
     [AvTransport] (多协议推送抽象层)
            ├──────────────────┬──────────────────┐
            ▼                  ▼                  ▼
     [WebRtcPusher]       [RtmpPusher]       [UdpSender]
      (FFmpeg whip)      (FFmpeg 封装)     (RTP 裸 UDP 包)
```

### 2.2 音频数据链路
```text
[Main Mic / Headset Mic] (采集源)
            │  (nau8822 Codec 采集)
            ▼
     [AudioCapture] (ALSA API 捕获，48000Hz 采样率)
            │  (S16_LE，按配置透传/降混/复制)
            ▼
     [AudioEncoder] (RTP/WebRTC/RTSP Opus 编码)
            │  (RTP 封包透传原始采集 PTS)
            ▼
     [AvTransport] (多协议分发)
```

## 3. 核心设计细节

### 3.1 视频非阻塞设计与丢帧逻辑
为保障音视频推送的最高实时性，系统摒弃了庞大的帧缓存。
- 视频队列容量设为 `2`，当采集线程向队列推帧时，如果发现队列已满，则**立即抛弃队列头部最老的一帧**，并将最新帧推入尾部。
- 视频硬解码/编码和网络发送模块在独立的 `VideoPipeline` 线程中轮询运行，最大限度保证采集线程出队动作的连贯。

### 3.2 WebRTC localhost DTLS 握手超时优化
- **痛点**：默认 Offer 采用 `a=setup:actpass` 使得服务器协商为 `active` 模式。在 localhost 环境下，服务器主动向客户端发送 DTLS 握手包往往被系统回环口路由所干扰，导致 DTLS 超时失败。
- **解决方案**：在 WHIP 客户端检测到服务器 IP 为本地或 loopback 地址时，强制调用 `config.bindAddress` 绑定至 `127.0.0.1` 物理回环；同时，在 SDP Offer 字符串中做动态替换，将 `a=setup:actpass` 强制覆写为 `a=setup:active`。这会使服务器成为 `passive` 接收方，强迫客户端主动发出 `ClientHello`。该处理将本地 WebRTC 的 DTLS 协商时延降到了 **10ms** 以内。

### 3.3 RTP over UDP 的空挂断兼容
- 修复了套接字返回 `ECONNREFUSED`（当播放器不在线时，内核返回 ICMP 端口不可达）引发管道断开崩溃的逻辑。在 `UdpSender::send` 中对该异常返回进行拦截和日志屏蔽，使得推流服务能够独立运行、随时挂起、随时上线接收。

## 4. 性能指标表现

经板端闭环测试，系统性能指标如下：
1. **13855 主路径**：V4L2 DMA-BUF 直通到 MPP EXT_DMA 编码输入，`前处理路径(fd/直通/拷贝)` 正常为 `0/30/0`。
2. **USB MJPEG 路径**：压缩 JPEG 输入 MPP MJPEG 解码，解码输出 NV12 DMA-BUF，`MJPEG(hw/sw/dma输出)` 正常为 `30/0/30`。
3. **MPP 视频编码延迟**：H.264/H.265 硬编码按 `编码` 字段单独统计，稳定维持 30 FPS。
4. **音频采集与推送**：20ms 帧稳定在 `50 ~ 51 fps`，日志 `耗时` 行输出 `排队=...ms 发送=...ms 总=...ms`。
5. **本地显示影响**：本地预览由 `debug.enable_preview` 控制；视频编码成功返回后移动同一个 `VideoFrame` 给 DisplayRenderer，DMA-only 帧通过 RGA fd 输入渲染，不再为桌面预览强制复制整帧 NV12。显示端仍需要 RGA 转 RGB/BGRA 和 X11/fb0 提交，开启后会增加少量 RGA/内存带宽占用。
6. **本地总延迟**：视频日志 `耗时` 行中的 `总=...ms` 表示采集 DQBUF 到 `send_video()` 返回的本地延迟，不等同于远端播放延迟。
7. **编码后封包收敛**：RTP/WebRTC 已改为逐包回调发送和 H.264/H.265 流式 NALU 扫描；RTMP/RTSP 使用内置 FFmpeg 8.1.1 写出协议封装；Opus/AAC 音频缓存使用 offset 消费，减少热路径搬移和临时分配。

## 5. 有待优化的地方 (Future Optimizations)

系统当前版本已经实现了高稳定性与良好的低延迟运行，但在未来的升级中仍有以下点有待进一步优化：

1. **编码后封包拷贝收敛**：
   摄像头到编码器主路径已经打通 DMA-BUF/MPP buffer 传递，RTP/RTMP/WebRTC/RTSP 封包热路径也已减少临时分配。剩余重点是评估 MPP 编码输出到 `EncodedPacket::data` 的 packet view 化，以及 WebRTC/SRTP 和 FFmpeg RTMP/RTSP 封装中仍不可避免的 CPU 可见 payload。
2. **动态码率自适应（Congestion Control）**：
   在网络发生波动、丢包或抖动时，目前的 RTMP/WebRTC/RTSP 仅以固定 CBR 码率硬推，容易造成网络拥塞 and 累积时延。后续可引入 RTCP 反馈解析模块，通过读取 `Receiver Report` 反馈的丢包率和 RTT 实时动态调整 MPP 编码器的 `Bitrate` 目标。
3. **Opus 自适应优化**：
   RTP、WebRTC 与 RTSP 已统一使用 Opus。后续可根据网络丢包率动态调整编码码率、FEC 和预期丢包参数。
4. **H.265 播放兼容性验证**：
   当前 H.265 已接入 MPP、RTP、WebRTC、RTSP 与 RTMP；RTMP 使用 FFmpeg 8.1.1 Enhanced RTMP/HEVC 封装。后续重点是验证不同播放器和服务端兼容性。
